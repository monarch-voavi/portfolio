Як я прискорив WordPress сайт у 3 рази без плагінів

8 хв читання· Volodymyr Avilov

Швидкість сайту — це не просто "приємний бонус", а реальний фактор, який впливає і на користувацький досвід, і на конверсію. На одному з проєктів я отримав типовий запит: сайт на WordPress був візуально нормальним, але завантажувався повільно, особливо на мобільних пристроях. Показник LCP тримався на рівні 4.2 секунди, а це вже відчутно б'є по сприйняттю сторінки.

Найцікавіше, що я свідомо не пішов шляхом "встановимо ще 5 плагінів оптимізації". Замість цього я вирішив спочатку знайти реальні причини повільної роботи. Аналіз показав кілька типових проблем: занадто великі зображення, блокуючі CSS та JavaScript-файли, непотрібні сторонні скрипти і завеликий DOM на головній сторінці.

Перший крок був очевидним — оптимізація медіа. Я перевів зображення у сучасні формати, зменшив їхній фактичний розмір і прибрав випадки, де браузер завантажував картинку у 2000 px для блоку, який фізично займав лише 400–500 px. Також я додав коректні width і height, щоб уникнути стрибків макета під час завантаження.

Другий крок — критичний CSS. Замість того щоб завантажувати весь стильовий пакет одразу, я виніс базові стилі для першого екрана в head, а решту підключив відкладено. Це зменшило час до першого візуального відображення й дало браузеру швидше показати користувачу основний контент.

Третій блок змін стосувався JavaScript. Я переглянув усі підключення теми та прибрав код, який взагалі не використовувався на конкретній сторінці. Частину скриптів перевів на defer, а деякі функції ініціалізував тільки після взаємодії користувача. Наприклад, елементи, які не впливали на перший екран, перестали блокувати рендеринг.

Окремо я звернув увагу на шрифти. Часто саме вони непомітно створюють затримку, особливо якщо підключено кілька накреслень і зовнішній CDN. Я скоротив кількість варіантів, локально підключив потрібні файли і налаштував font-display: swap, щоб текст не зникав під час завантаження шрифтів.

Ще один важливий момент — сторонні сервіси. Аналітика, чати, віджети, кнопки соцмереж — усе це здається дрібницями, поки не подивишся waterfall у DevTools. Я залишив тільки те, що справді давало цінність бізнесу, а зайві інтеграції прибрав або відклав до моменту взаємодії.

У результаті LCP вдалося знизити з 4.2 секунди до 1.4 секунди. І головне — це було досягнуто без WP Rocket, без Autoptimize і без "магічної кнопки оптимізації". Висновок простий: перш ніж ставити плагіни, варто розібратися, що саме гальмує сайт. Часто найбільший приріст дає не автоматизація, а точкові технічні рішення.

Site speed isn't just a nice-to-have — it's a real factor that directly impacts user experience and conversion. On one of my projects, I got a typical request: the WordPress site looked fine, but loaded slowly, especially on mobile. LCP sat at 4.2 seconds, which is noticeably painful.

What's interesting — I consciously avoided the "install 5 more optimization plugins" route. Instead I decided to find the actual causes of the slowness first. Analysis showed typical issues: oversized images, blocking CSS and JavaScript files, unnecessary third-party scripts, and a too-heavy DOM on the landing.

First step was obvious — media optimization. I converted images to modern formats, reduced their actual size, and eliminated cases where the browser was loading a 2000px image for a block that only occupied 400–500px. I also added proper width and height attributes to prevent layout shift.

Second step — critical CSS. Instead of loading the entire stylesheet upfront, I extracted base styles for the first screen into the head and loaded the rest deferred. This reduced time-to-first-paint and let the browser render primary content faster.

Third block of changes targeted JavaScript. I reviewed all theme scripts and removed code that wasn't actually used on the specific page. Some scripts moved to defer, and certain functions were initialized only after user interaction. Elements that didn't affect the first screen stopped blocking rendering.

I also paid special attention to fonts. They often quietly create delay, especially when multiple weights are loaded from an external CDN. I reduced weight variants, self-hosted the necessary files and set font-display: swap so text wouldn't disappear during font load.

Another important point — third-party services. Analytics, chats, widgets, social buttons — it all feels like small stuff until you see the waterfall in DevTools. I kept only what genuinely delivered business value and removed or deferred the rest until user interaction.

The result — LCP dropped from 4.2s to 1.4s. And the key thing: this was achieved without WP Rocket, without Autoptimize, without any "magic optimization button". The conclusion is simple: before installing plugins, figure out what's actually slowing the site down. Often the biggest gain comes not from automation, but from targeted technical decisions.

Server Actions у Next.js 14: що я дізнався за пів року

12 хв читання· Volodymyr Avilov

Коли Server Actions тільки почали активно обговорювати, вони виглядали майже як універсальна заміна звичним API routes. Ідея дуже приваблива: викликати серверну логіку прямо з компонента, без зайвого шару API, без дублювання типів і без складної ручної обробки fetch-запитів. Але після кількох місяців реального використання картина стала більш приземленою.

Мій головний висновок такий: Server Actions — це сильний інструмент, але не повна заміна всього підряд. Найкраще вони працюють у сценаріях, де є тісний зв'язок між формою, мутацією даних і оновленням UI. Наприклад, створення запису, оновлення профілю, зміна статусу або відправка форми з подальшою revalidation — тут досвід справді стає простішим.

Що мені сподобалось найбільше — менше шаблонного коду. Не потрібно окремо створювати endpoint, писати fetch, описувати JSON-відповідь і дублювати перевірки на клієнті та сервері. Якщо логіка невелика й добре прив'язана до конкретної сторінки, Server Actions відчуваються природно і дуже швидко в розробці.

Але є й нюанси. Перший — межа між клієнтським і серверним кодом стала менш очевидною. Якщо команда ще не звикла до архітектури App Router, легко заплутатися, де саме виконується логіка, як передаються дані і чому щось раптом не працює в браузері. Для маленьких задач це не проблема, але в більшому проєкті без дисципліни починається хаос.

Другий момент — не все треба переносити в Server Actions. Якщо у вас публічний API, інтеграція з мобільним застосунком, webhooks або зовнішні клієнти, класичні API routes залишаються логічнішим рішенням. Вони краще підходять там, де потрібен незалежний контракт між клієнтом і сервером.

Ще одна практична річ — валідація. Server Actions не скасовують необхідність перевіряти вхідні дані. Навпаки, через відчуття "простоти" є ризик почати писати мутації занадто легковажно. У моєму випадку найкраще спрацювала зв'язка з Zod: вона дозволяє тримати схему явно, повертати зрозумілі помилки і не змішувати бізнес-логіку з хаотичними if-перевірками.

З точки зору продуктивності та DX я б сказав так: Server Actions добре прибирають зайву складність у CRUD-сценаріях, але не є універсальним патерном для всіх випадків. Якщо сторінка тісно пов'язана з серверною мутацією — це чудовий вибір. Якщо потрібна незалежна API-архітектура — краще не ламати те, що вже добре працює.

Після пів року використання мій підхід став простішим: я не питаю "чим замінити API routes", а питаю "де Server Actions справді спрощують код". І в цьому формулюванні вони приносять найбільше користі.

When Server Actions first started getting buzz, they looked almost like a universal replacement for traditional API routes. The idea is really appealing: call server logic directly from a component, without an extra API layer, type duplication, or complex manual fetch handling. But after several months of real usage, the picture became more grounded.

My main conclusion: Server Actions are a powerful tool, but not a full replacement for everything. They work best in scenarios with a tight coupling between a form, data mutation, and UI update. Creating a record, updating a profile, status change, or form submission with follow-up revalidation — here the experience really does become simpler.

What I liked most — less boilerplate. No need to separately create an endpoint, write fetch, describe the JSON response and duplicate validation on both client and server. If the logic is small and well-tied to a specific page, Server Actions feel natural and very fast to develop with.

But there are nuances. First — the boundary between client and server code became less obvious. If a team isn't used to App Router architecture yet, it's easy to get confused about where logic runs, how data is passed, and why something suddenly doesn't work in the browser. For small tasks that's not a problem, but in larger projects without discipline, chaos begins.

Second point — not everything should be moved to Server Actions. If you have a public API, mobile app integration, webhooks or external clients, classic API routes remain the more logical choice. They fit better where you need an independent contract between client and server.

Another practical thing — validation. Server Actions don't eliminate the need to check input data. If anything, the feeling of "simplicity" creates risk of writing mutations too carelessly. For me the best combo was with Zod: it lets you keep the schema explicit, return clear errors, and avoid mixing business logic with chaotic if-checks.

From performance and DX perspective I'd put it like this: Server Actions nicely remove excess complexity in CRUD scenarios, but aren't a universal pattern for every case. If a page is tightly tied to a server mutation — great choice. If you need independent API architecture — better not break what already works.

After six months of use my approach got simpler: I don't ask "what replaces API routes" anymore, I ask "where do Server Actions actually simplify the code". In that framing they bring the most value.

Мій dev setup 2026: від Brave до Claude Code

6 хв читання· Volodymyr Avilov

Ідеальний dev setup для мене — це не набір гучних назв, а система, яка щодня прибирає зайве тертя з роботи. З часом я перестав гнатися за "наймоднішими" інструментами й залишив тільки ті, що реально дають швидкість, фокус і передбачуваність у щоденному workflow.

Почну з браузера, бо саме він у розробника відкритий майже постійно. Я перейшов на Brave не через хайп, а через практичність: у ньому з коробки є блокування реклами й трекерів, підтримка Chrome-розширень, а сам браузер робить сильний акцент на приватності та продуктивності.

Для моєї роботи це означає дуже просту річ: менше шуму, менше сміття на сторінках і менше залежності від десятка окремих розширень. Brave також просуває вбудованого AI-асистента Leo, що добре лягає в сучасний робочий процес, коли треба швидко пояснити сторінку, переформулювати текст або розібратись у контексті без постійного стрибання між сервісами.

Ще одна причина, чому Brave добре вписався в мій setup, — відчуття "робочого браузера", а не експериментального середовища. Якщо коротко, мені потрібен інструмент, який стабільно тягне щоденні задачі: документацію, DevTools, кілька проєктів, адмінки, GitHub, таск-трекер, пошту й купу вкладок одночасно. Brave тут відчувається надійним компромісом між швидкістю, сумісністю і контролем над приватністю.

Другий ключовий елемент мого setup — Claude Code. Я не сприймаю його як "чарівну кнопку, що пише продукт", зате дуже ціную як сильного асистента в рутині: швидко розібрати чужий код, запропонувати рефакторинг, допомогти з чернеткою функції або підсвітити слабкі місця в логіці.

Найкраще така зв'язка працює тоді, коли кожен інструмент має чітку роль. Brave закриває щоденний веб-шар роботи — пошук, документацію, тестування, адмінки, AI у браузері, — а Claude Code допомагає не застрягати в дрібних технічних вузлах і швидше рухатись по коду.

У підсумку мій dev setup у 2026 році став значно простішим. Не "більше інструментів", а "менше зайвого": Brave як основний браузер для швидкої й чистої роботи та Claude Code як практичний AI-шар поверх реальних задач розробника.

The ideal dev setup for me isn't a collection of buzzwords — it's a system that removes daily friction from work. Over time I stopped chasing the "trendiest" tools and kept only those that deliver real speed, focus and predictability in everyday workflow.

Let's start with the browser, since for a developer it's open almost constantly. I switched to Brave not because of hype, but for practicality: it has ad and tracker blocking out of the box, supports Chrome extensions, and the browser itself puts strong emphasis on privacy and productivity.

For my work this means a simple thing: less noise, less junk on pages, and less dependency on a dozen separate extensions. Brave also promotes its built-in AI assistant Leo, which fits well in a modern workflow where you need to quickly explain a page, rephrase text, or understand context without constantly jumping between services.

Another reason Brave fit well into my setup — the feel of a "working browser" rather than an experimental environment. Put simply, I need a tool that reliably handles daily tasks: docs, DevTools, several projects, admin panels, GitHub, task trackers, email, and tons of tabs at once. Brave feels like a reliable compromise between speed, compatibility, and privacy control here.

The second key element of my setup is Claude Code. I don't see it as a "magic button that writes the product", but I really value it as a strong assistant for routine tasks: quickly parse someone else's code, suggest refactoring, help draft a function, or highlight weak spots in logic.

This combo works best when each tool has a clear role. Brave covers the daily web layer — search, docs, testing, admin panels, in-browser AI — and Claude Code helps me not get stuck in small technical knots and move through code faster.

Overall, my dev setup in 2026 became much simpler. Not "more tools" but "less fluff": Brave as the main browser for fast clean work, and Claude Code as a practical AI layer on top of real developer tasks.
